# `sweny new <id>` — Marketplace Install

**Date:** 2026-04-14
**Status:** Design — awaiting review
**Supersedes partially:** `2026-04-05-workflow-marketplace-design.md` (consume side only; publish / site UI untouched)

## Summary

Extend `sweny new` so it can install a workflow directly from the public marketplace repo (`swenyai/workflows`) without any wizard flow:

```
sweny new pr-review          # fetch workflows/pr-review.yml, adapt, install
```

When the workflow's declared providers don't match what the user has in `.sweny.yml`, the CLI invokes the existing `refineWorkflow()` (Claude) to rewrite the workflow for the user's setup, renders the resulting DAG, and runs the existing accept/refine/cancel loop. If no agent is configured, install as-is with a warning.

No new command. No cloud dependency. No new skill metadata. This is a new branch inside `runNew()` that reuses every primitive the wizard already uses.

## Why

- Publish side is built (`sweny publish`) but consume side is not — the loop is broken.
- Hardcoded templates cap what `sweny new` can scaffold at four workflows.
- External consumers (docs, Claude skills, landing pages) need a "copy-paste install" primitive: `sweny new <id>` is that.
- Existing `refineWorkflow()` + per-skill credential derivation + idempotent file writes already solve 90% of the install mechanics. The missing piece is a GitHub raw fetch and a small router on `sweny new`'s arg.

## Scope

### In
- `sweny new <id>` — fetch a workflow from `github.com/swenyai/workflows` raw, adapt if needed, install
- Adaptation via `refineWorkflow()` when the workflow's declared skills mismatch the user's `.sweny.yml` providers
- Fallback: install as-is with a warning when no agent is configured
- `sweny new` (no args) — existing wizard gets one new option: "Browse marketplace" (lists workflows from the repo; picking one delegates to the same install path)
- Error handling: 404, rate limit, malformed YAML, network failure
- Tests: mocked `fetch`, mocked `ClaudeClient`, covering happy path + mismatch + no-agent fallback

### Out
- Standalone `sweny install` command (explicitly rejected — `sweny new <id>` is the only surface)
- Cloud API proxy (raw GitHub only for v1; cloud proxy is a future optimization for search/ranking)
- Local caching beyond the OS HTTP cache (worry about this only if rate limits bite)
- Versioning / pinning workflow revisions (`sweny new pr-review@v2`) — future
- Publishing flow changes (existing `sweny publish` untouched)
- Marketplace web UI (`marketplace.sweny.ai`) — out of scope, owned by the earlier design spec
- Skill compatibility metadata / deterministic provider-swap tables — explicitly rejected

## UX

### `sweny new <id>` (direct install)

```
$ sweny new pr-review
◇  Fetching pr-review from swenyai/workflows…
◇  Workflow uses: github, linear
◇  Your project: github, github-issues
◇  Adapt workflow to use github-issues instead of linear?  (Y/n) y
◇  Refining via Claude…

   [rendered DAG diff]

◇  Looks good?  Yes / Refine / Cancel
◇  Created .sweny/workflows/pr-review.yml
◇  Appended 1 new key(s) to .env

Next steps:
  1. Fill in your API keys in .env
  2. sweny workflow run .sweny/workflows/pr-review.yml
```

### `sweny new pr-review` with no `.sweny.yml` (fresh project)

Same flow, but `.sweny.yml` is created from the workflow's inferred providers (existing `buildSwenyYml()` path), and no adaptation is needed because there's nothing to mismatch against.

### `sweny new` (no args — existing wizard)

Picker gains one option at the top:

```
What do you want to do?
❯ Browse marketplace          (fetch and install a published workflow)
  PR Review
  Issue Triage
  Security Scan
  Release Notes
  End-to-end browser testing
  Describe your own
  Start blank
```

Selecting "Browse marketplace" fetches the workflow index (`workflows/index.json`, see below), shows a searchable list, and on selection delegates to the same install path as `sweny new <id>`.

### No-agent fallback

If `ANTHROPIC_API_KEY` isn't set and a mismatch is detected:

```
◇  Workflow uses: linear. Your project uses: github-issues.
◇  Adaptation needs an agent (ANTHROPIC_API_KEY). Installing as-is.
◇  Created .sweny/workflows/pr-review.yml

⚠  This workflow references `linear`. To run it, either:
   - add LINEAR_API_KEY to .env, or
   - edit .sweny/workflows/pr-review.yml to use github-issues
```

## Marketplace repo layout (`swenyai/workflows`)

Flat:

```
workflows/
  pr-review.yml
  issue-triage.yml
  security-scan.yml
  release-notes.yml
  ...
index.json                 # generated, lists {id, name, description, skills} for browse
README.md                  # how to submit
```

- **`<id>`** resolves to `workflows/<id>.yml` on the `main` branch.
- **`index.json`** is generated by CI on merges to `main` (simple script that walks `workflows/*.yml`, parses frontmatter/metadata, emits JSON). The CLI only fetches `index.json` for the "Browse marketplace" wizard branch — the direct `sweny new <id>` path hits the YAML file directly without touching the index.
- Workflow file format is the existing `.sweny/workflows/*.yml` schema — no marketplace-specific envelope.

## Adaptation mechanism

The "adapt workflow to user's system" step is:

1. `extractSkillsFromYaml(marketplaceYaml)` → `["github", "linear", "datadog"]` (existing function, no changes).
2. Read user's `.sweny.yml` → `{ sourceControl: "github", issueTracker: "github-issues", observability: null }`.
3. Compute mismatch: any skill in the workflow whose role (issue-tracker / observability / source-control / notification) binds to a different provider than the user has, OR a role the user hasn't configured at all.
4. If mismatch is empty → install as-is, done.
5. If mismatch is non-empty AND agent is available:
   ```ts
   const adaptPrompt = `This workflow uses ${mismatch.workflow}. The target project uses ${mismatch.user}. Rewrite the workflow to use the target's providers while preserving intent.`;
   const adapted = await refineWorkflow(parsedWorkflow, adaptPrompt, { claude, skills, logger });
   ```
   Then the existing accept/refine/cancel loop from `runCustomWorkflowBuilder` (`new.ts:735–779`) takes over.
6. If mismatch is non-empty AND no agent → install as-is, print warning (see UX).

"Role" here means the abstract slot. Skills already declare `category: "git" | "observability" | "tasks" | "notification" | "general"` (`types.ts:39`). `computeProviderMismatch` uses that field directly, mapping:

| `skill.category` | `.sweny.yml` key |
|---|---|
| `git` | `source-control-provider` |
| `observability` | `observability-provider` |
| `tasks` | `issue-tracker-provider` |
| `notification` | `notification-provider` |
| `general` | — (skipped; no provider slot) |

A mismatch is: workflow references a skill with `category: "tasks"` whose `id` differs from the user's `issue-tracker-provider` (with the same mapping-swap rule for each category).

## File changes

**`packages/core/src/cli/main.ts`** — arg routing. Pass the positional `<id>` arg through to `runNew`.

**`packages/core/src/cli/new.ts`**:
- `runNew(options?: { marketplaceId?: string })` — new optional arg. When set, skip the wizard picker and go straight to the install path.
- New function `fetchMarketplaceWorkflow(id: string): Promise<{ yaml: string; id: string }>` — raw GitHub fetch, typed error on 404/ratelimit/network.
- New function `computeProviderMismatch(workflowSkills: string[], existingConfig: FileConfig, availableSkills: Skill[]): Mismatch[]` — pure function, testable.
- New function `adaptWorkflowForProject(workflow, mismatch, opts): Promise<WorkflowTemplate | null>` — thin wrapper that builds the adapt prompt and calls `refineWorkflow`, then runs the existing accept/refine/cancel loop. Reuses `DagRenderer`.
- Wizard picker: add `{ value: "__marketplace", label: "Browse marketplace", ... }` at the top. Selecting it calls `fetchMarketplaceIndex()` → `p.select()` → delegates to install path.
- `fetchMarketplaceIndex(): Promise<MarketplaceEntry[]>` — fetches `index.json`, parses, returns.

**`packages/core/src/cli/new.test.ts`** — add cases:
- `sweny new <id>` happy path (no mismatch, install as-is)
- `sweny new <id>` with mismatch + agent → `refineWorkflow` invoked + DAG rendered + accept path
- `sweny new <id>` with mismatch + no agent → warning printed, install as-is
- `fetchMarketplaceWorkflow` error handling: 404 → typed error with helpful message; rate limit → suggest retry; network failure → suggest `--offline` (future) or retry
- `computeProviderMismatch` unit tests for each role (source-control, issue-tracker, observability, notification)

**`packages/core/src/cli/templates.ts`** — no changes. The built-in templates remain as the "offline fallback" list.

**No changes to** `buildWorkflow` / `refineWorkflow` / `ClaudeClient` / credential collection / `.env` writing / `.sweny.yml` writing. All existing.

## Error handling

| Failure | Behavior |
|---|---|
| 404 on `<id>.yml` | `Workflow "pr-review" not found in swenyai/workflows. See https://marketplace.sweny.ai for available workflows.` |
| GitHub rate limit (403 with `X-RateLimit-Remaining: 0`) | `GitHub rate limit hit. Retry in <reset time>, or set GITHUB_TOKEN to raise the limit.` |
| Network failure | `Could not reach github.com. Check your connection and retry.` |
| Malformed workflow YAML | `Downloaded workflow is malformed YAML. This is a bug in the marketplace — please file an issue.` + paste the raw content in `~/.sweny/cache/bad-download-<id>.yml` for debugging |
| `refineWorkflow` throws | Log the error, fall back to install-as-is path with warning |
| User cancels adapt loop | Exit with `Setup cancelled.` — do not install partial |

## Testing

All tests go in the existing Vitest suite. Pattern matches existing CLI tests:
- Mock `global.fetch` with `vi.spyOn(globalThis, "fetch").mockImplementation(...)` (per auto-memory, `mockImplementation` not `mockResolvedValue`).
- Mock `ClaudeClient` + `refineWorkflow` for the adaptation tests.
- Use `@clack/prompts` mocks as other `new.test.ts` cases do.

Coverage target: parity with existing `new.ts` coverage. The pure functions (`computeProviderMismatch`, `fetchMarketplaceWorkflow`) get 100% line coverage; the interactive glue is thin and tested via the existing prompt-mock pattern.

## Rollout

1. Land `sweny new <id>` + tests in `@sweny-ai/core`.
2. Create `swenyai/workflows` repo with 4 workflows (the current built-in templates) as seeds.
3. Add `index.json` generator (GitHub Action on push to `main`).
4. Update docs site with "Install a workflow: `sweny new <id>`" as the primary call-to-action.
5. Add the "Browse marketplace" wizard option once the repo has ≥4 workflows.

Phases 1–2 are the MVP; 3–5 can follow iteratively.

## Out of scope / future

- **Pinning**: `sweny new pr-review@v2` or `@sha1234` — would require tags/commits in the marketplace repo; defer.
- **Private marketplaces**: org-level forks (`sweny new mycorp/pr-review`) — defer; would route through a different resolver.
- **Local caching**: HTTP cache is fine for v1. If rate limits bite, add `~/.sweny/cache/marketplace/` with TTL.
- **Cloud proxy** (`api.sweny.ai/marketplace/<id>`): value is search/ranking/telemetry, not latency. Add when marketplace has >50 workflows and search matters.
- **Workflow-specific adapter prompts** (per-workflow `skill.md` with custom adaptation instructions): interesting, but the generic `refineWorkflow` prompt works for v1. Revisit if we see the LLM making bad adaptation choices in practice.
